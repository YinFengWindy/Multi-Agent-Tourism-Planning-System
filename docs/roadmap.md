# 实施路线与验收标准

## 1. 分阶段实施路线

### Phase 0：架构基座

目标：完成最小闭环基础设施与工程骨架。

- 搭建 `API Gateway`、`Planner Service`、`MCP Gateway`
- 接入 Redis、MongoDB、JWT、Rate limiting
- 完成计划创建、进度流式输出和快照存储

验收标准：

- 可提交一条旅游规划请求
- 可返回规划中的流式状态
- MongoDB 可保存计划和快照
- Redis 可用作缓存与限流

### Phase 1：核心三 Agent 落地

目标：形成可用旅游规划最小业务价值。

- 实现 `Transport Agent`
- 实现 `Hotel Agent`
- 实现 `Route Agent`
- 打通 `Plan-Execute-Replan` 基本链路

验收标准：

- 给定城市、日期、预算后能返回可执行的 2~4 天行程
- 交通、住宿、日程三类结果可互相约束
- 预算超限或时间冲突时可触发至少一次重规划

### Phase 2：补齐预算与风险治理

目标：提升结果质量与稳定性。

- 实现 `Attraction Agent`
- 实现 `Budget Agent`
- 实现 `Risk Agent`
- 引入缓存、语义摘要和工具兜底机制

验收标准：

- 预算拆分完整
- 可识别天气或闭馆风险
- 工具超时时可从缓存回退
- 平均响应速度较无缓存方案明显提升

### Phase 3：生产化增强

目标：支撑真实流量与持续迭代。

- 完成监控、告警、审计与权限体系
- 增加多版本对比、历史回溯、分享导出
- 建立压测、灰度发布和成本看板

验收标准：

- 支持多实例部署
- 有关键链路观测与告警
- 可查看规划历史与重规划原因
- 可量化缓存命中率与工具成功率

## 2. 非功能指标建议

### 2.1 性能

- 常规规划请求 P95 小于 8 秒
- 命中缓存的常见查询 P95 小于 1.5 秒
- 单次规划中工具调用成功率目标大于 95%

### 2.2 质量

- 规划结果包含主方案 + 至少一个备选方案
- 时间冲突检测覆盖率 100%
- 预算拆分完整率 100%

### 2.3 稳定性

- 单个工具异常不导致整条链路直接失败
- Redis 或 MongoDB 单点异常时有明确降级路径

## 3. 团队协作建议

- 后端 1：API Gateway、鉴权、会话和外部接口
- 后端 2：Planner Service、LangGraph 状态图、重规划
- 后端 3：MCP Gateway、工具适配器、缓存与工具治理
- 算法 / Agent：提示词、状态设计、领域 Agent 策略
- 前端：规划表单、流式展示、结果页、版本对比

## 4. 交付优先级

优先顺序建议：

1. API Gateway
2. Planner Service
3. MCP Gateway
4. Transport / Hotel / Route 三个核心 Agent
5. 前端规划页与结果页
6. 预算、风险、缓存、监控
