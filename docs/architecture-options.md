# 架构方案对比

## 1. 目标

为了满足高并发、可重规划、可扩展和工具标准化接入的需求，本系统在设计阶段对三种主流架构进行了对比。

## 2. 方案候选

### 方案 A：中心编排 + 分布式领域 Worker（推荐）

- `Planner Service` 统一承载 Parent Agent 与 LangGraph 状态机
- 领域子 Agent 以独立 Worker 形式部署，可按能力横向扩容
- 所有工具统一经过 `MCP Gateway`
- 适合多 Agent 并行协同、重规划和审计追踪

### 方案 B：完全对等的 Agent Mesh

- 各 Agent 之间直接协商，不依赖中心编排器
- 理论上自治性更高，但消息链路复杂，冲突处理成本高
- 不利于旅游规划场景中的约束一致性和结果审计

### 方案 C：单体编排服务

- 所有 Agent、工具调用与 API 入口都集中在一个服务内
- 开发成本最低，适合原型验证
- 但横向扩展差，工具治理和资源隔离能力不足

## 3. 对比表

| 维度 | 方案 A | 方案 B | 方案 C |
| --- | --- | --- | --- |
| 并行协作 | 强 | 中 | 弱 |
| 可观测性 | 强 | 弱 | 中 |
| 重规划控制 | 强 | 中 | 中 |
| 开发复杂度 | 中 | 高 | 低 |
| 运维复杂度 | 中 | 高 | 低 |
| 工具治理 | 强 | 中 | 弱 |
| 适合生产级旅游系统 | 是 | 否 | 仅限 MVP |

## 4. 结论

推荐使用 **方案 A：中心编排 + 分布式领域 Worker**。

原因如下：

- 旅游规划天然依赖“全局约束一致性”，中心编排更适合处理预算、时间窗、地理路径等交叉约束
- 子 Agent 可以独立扩展，适合高峰期对交通、酒店、路线等热点能力做弹性扩容
- MCP Gateway 作为工具统一出口，便于做协议标准化、缓存、限流和容错
- Parent Agent 统一负责 `Plan-Execute-Replan`，便于实现可解释、可审计的规划过程

## 5. 最终选型

最终采用：

- 编排模式：中心编排
- 任务通信：`Redis Streams`
- 工具协议：`MCP`
- 数据存储：`MongoDB + Redis`
- 交互层：`Vue 3 + API Gateway + SSE`
